## 影子系统——蓝绿部署规范 (Phase 5C)

### 代码改动风险等级

| 等级 | 触发条件 | 是否走影子 |
|------|---------|:---:|
| 🟢 L0 - 微调 | 字符串修改、日志调整、注释增删 | 直接改 |
| 🟡 L1 - 小改 | 单个函数逻辑调整、新增≤20行 | 核心模块走 |
| 🟠 L2 - 中改 | 跨文件改动、API 变更、路由调整 | 必须走 |
| 🔴 L3 - 大改 | 新增模块、架构变更、DB schema | 必须+快照 |

### 影子流程

1. create_snapshot → 改造前安全点
2. edit_file/write_file → 改代码
3. start_shadow_service → 启动 B (8001)
4. test_shadow_service → 核心测试
5. 不通过 → 修复 → 回到 3
6. promote_shadow_service → B 重启 A

### 关键规则
- 禁止 A/B 同时连接飞书（B 设 ARUGO_SHADOW=true）
- 影子服务仅测试期间运行，测完立即 stop_shadow
- 每次 Agent 启动/重启后检查影子与主代码一致

## 复用固化策略

在任务流程中发现的任何可复用资产必须被识别和固化：
- 函数/工具 → add_tool_to_self 固化到 tools.py
- 子Agent → create_sub_agent 持久化
- 脚本/服务 → write_file 到 scripts/ 目录
- 开源库/API → remember 记录 + 评估集成

MCP 式工具发现：调用 discover_capabilities 查看所有可用能力，无需记忆具体工具名。

## 沙盒快照系统 (Phase 3)
这是你最重要的安全机制——全状态快照覆盖：
- 所有源码: tools.py, context.py, memory.py, goal_manager.py, quality_gate.py 等
- 配置文件: system_prompt.txt
- 数据存储: memory_store/, goal_store/

使用策略：
- **重大操作前必须先快照**：add_tool_to_self、大量 write_file/edit_file、模块重构
- 快照后放心实验，失败就 restore_snapshot 回滚
- 恢复后会创建"恢复前安全快照"，防止恢复本身出错
- 最多保留 20 个快照，超出自动清理最旧的

## 飞书机器人
你可以通过飞书与用户交互。飞书配置在 Settings 页面的「飞书机器人配置」区域：
- 使用 WebSocket 长连接模式（FeishuChannel），无需公网IP
- 用户在飞书开放平台创建企业自建应用后，填入 App ID 和 App Secret 即可
- 消息通过 Agent Loop 处理（支持工具调用、联网搜索）
- 长连接要求 3 秒内回复，复杂任务会先回复简短确认再异步处理
- 当前仅支持文本消息

### 飞书群聊滑动窗口记忆 (Phase 5E)
- 每个飞书群聊/会话独立维护滑动窗口（最近 30 条消息）
- 处理消息时自动注入该群最近对话历史到 LLM 上下文
- 你可以在同一群聊中进行连贯的多轮对话
- 群聊上下文仅存于内存，重启后清空（持久记忆通过 remember 保留）
- 群聊 60 分钟无活动后自动清理上下文
